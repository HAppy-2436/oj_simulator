# OJ Simulator 源码及架构详细分析文档

该文档通过逐行分析 `gui_app.py` 的源代码结构，对整个 OJ 自动化刷题项目进行了详尽的梳理和分类。这份文档清晰描述了系统的方方面面，包括组件构建、用户配置管理、网页自动化抓取/提交逻辑，以及后台任务调度的状态机。

---

## 1. 项目概览

**名称**: OJ 拟人化刷题模拟器 (OJ Simulator)
**作用**: 为用户自动提取某个 Online Judge 上的题目，将题目数据转发给设定好的大模型（如 DeepSeek/GPT），并将生成的代码自动填写回网页端提交，如果没跑通过，甚至可以继续纠正重试。
**核心依赖**:
- `tkinter` 构建图形界面及核心控制事件。
- `playwright` 驱动浏览器完成登录、代码操作和结果验证（支持 Headless 模式运行）。
- `requests` 用于与 LLM (大预言模型) API 交互。
- 引入多线程 (`threading`) 与倒计时计划控制每日打卡机制。

---

## 2. 界面与控制器架构

项目依托于 `SimulatorGUI` 主类管理。该类在初始化 (`__init__`) 时将界面切割为三个功能区。

### A. 主控区 (Frame Main Control)
- **设置控制**: 提供“⚙️ 设置”面板的开发展现切换。
- **模式选项**:
  - `无头模式`: (Headless) 决定 Playwright 是否静默在后台操作。
  - `极速调试`: 跳过复杂的拟人漫长停顿延迟（`time.sleep`）。
  - `展开LLM日志`: 在右下角控制台印出发送给 GPT/Deepseek 的源生请求和响应。
  - `系统托盘模式`: (基于 Pystray) 实现应用点❌时仍躲藏在后台刷题。
- **任务动作**: [重置]、[开始]、[停止]、[补漏检查 (Retro Check)] 循环触发机制。

### B. 设置面板容器 (Settings Container)
收纳自 `_build_settings()` 注册生成的一系列数据与选项调整项，并提供“保存配置”将此页面数据转存至 `config.json`:
- **基础与大模型设置**: OJ 地址、账号密码、API Token、BaseURL。
- **目标与睡眠设置**: 可以设置“每日完成额数量(波动作差)”、“开始工作和停止工作休眠的时间段”。
- **频率与重试参数**: 设置从答题、敲字到思考时延时 (`read_delay`, `write_delay`)；遭遇解答不合格(WA)时的破防延时；和针对多次出错设置最高出错限制次数。
- **代码风味**: 可使用自定义自然语言提示 (`custom_prompt` 变量名如 `多用单字母, 使用入门 STL, Qt代码规范`) 约束模型输出的代码。
- **高难判断策略**: 基于对字符长度或获取的题目 Difficulty Mapping，判断题目高难后执行“延时双倍时间”或“跳过并进入下一题”。

### C. 运行日志区 (Log Display)
依靠 Text Widget 显示各时段详细进展，如 `[完成/题目序号]`，`检测到验证码，准备捕获`等状态更新。依靠 `root.after` 以及线程隔离实现控制台滚动。

## 3. 持久化数据机制

代码重度依赖本地的文件 I/O 控制缓存状态：

- **`config.json`**:
  存放 `_build_settings()` 中提及过的各种 URL、账号密码及各类延时配置，由 `save_config()` 和 `load_config()` 打包读写。

- **`state.json`**:
  受 `load_or_init_daily_state()` 支配。保存每天刷题进度的逻辑：
  - 如果检测到当天换日 (比如到了每天 `4:00 AM`)，系统自动借助随机函数（目标基准值 ± fluctuation），推算生成新一天的**目标数量** (`target_count`)。
  - 并维护当前已执行完毕 `completed` 的次数。

- **`difficulty.json`**:
  通过后台爬虫读取带有 `/contest/` 的竞赛页面表格页，抓取指定题型的“绿牌/标签难度并缓存”。

## 4. 自动化核心与重头戏：任务运行器 (`run_bot`)

项目功能运转的最核心在 `run_bot` 方法内，该方法开启了隔离出来的子线程跑死循环，完全交由 Playwright 管理 DOM：

**Phase 1. 初始化驱动与登录**:
- 判断用户配置提取要刷的下一道 URL ID (或倒退遍历，调用 `decrement_url` 向前查漏补缺)。
- 从 Playwright 启动 Chromium，跳转登录页。匹配元素 `#id_username` 注入账号与密码。
- **对抗验证码 (Captcha)**: 截取包含 `captcha` 元素的图片弹窗，在主线程展示让用户人工解析输入识别码或跳过。

**Phase 2. 调度控制引擎 (`While completed < target`)**:
这部分不仅是循环迭代判断做题，还有休息期的判断：如果系统判断系统时间越过了设定的区间点，则进入强制循环中断补眠 (基于事件比对 `time.time() < wakeup_time`)，直达次日唤醒阶段。

**Phase 3. 题目抽取并进行作答判定**:
- **拉题与剥离判责**:
  拉取题目中的 Description / 难度特征 (`.twelve.wide.column`)。若该题早已有人写过内容而且带存在 "已通过(Accepted) 绿灯"，则直接触发 `self.increment_url` 跳下一题。
- **指令重造与 LLM 通讯 (`call_llm_api`)**:
  获取完 HTML，利用 Regex 排空无用标签后加入 Custom Prompt 发送给大模型生成 C++ 代码，通过获取 JSON 里的 `content` 将回答去除前缀头段后交出。
- **拟人填涂操作 (`inject_and_submit`)**:
  聚焦编辑框，按压退格控制清空 `Meta/Control + A` 后键入新代码。自动向下滚动页面并匹配 `[type='submit']` 或 `button.green` 发出评测请求。

**Phase 4. 裁判端反馈 (`wait_for_judgment_result`)**:
- **循环判定**: 代码交出后陷入死等，查询页面 `.status` 等区域状态如果为 `queue` / `compiling` 继续卡着睡。当出现了 `green` / `red` 标识才打破返回。
- 遇错（Wrong Answer）记录，进入纠正重修循环，甚至让大模型分析报错回显并再次修正填回直至达到 `max_retries` 破防弃题。

## 5. 项目潜在不足和存在问题分析 (重构驱动力)

1. **强耦合重灾区**: UI操作 (`self.entry_user.get()`) 、IO存储逻辑 (`json.load(f)`) 和核心 Playwright 的抓取爬虫被挤在同一个几百行的方法内，一旦有一端报错全部牵连停止。
2. **逻辑结构太深**: `run_bot` 中的 `try/except` 及 `while` 嵌套曾达到五六层级之深，对于缩进极易破坏而产生 `NameError` 和 `SyntaxError`（如 `is_ac_already` 的局部跳穿）。
3. **缺少弹性**: 在出现页面DOM元素更换名，如隐藏验证码等标签无法被精确定位的情况时整个应用将报错进入死点。
